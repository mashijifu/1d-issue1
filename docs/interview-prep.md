## README とこのドキュメントを同じにすべきか

**同じにする必要はありません。役割を分けるのがおすすめです。**

| 観点 | README | このドキュメント（interview-prep） |
| --- | --- | --- |
| 読者 | 採点者・リポジトリを初めて見る人 | 自分（面談前の復習） |
| 分量 | 短く、一覧で読める | 長くてもよい、話す用のネタ |

運用のコツは次の2点です。

1. **README** … 「何をしたか」「なぜそうしたか（要約）」が分かればよい。ページネーションは **2〜4行** あれば十分。
2. **このドキュメント** … 面談で聞かれそうな **「なぜ無限スクロールではないか」「Route Handlerの意味は」** などを深掘りする。

内容が食い違わないよう、**READMEの要約はこのドキュメントの「結論」と一致**させる（矛盾だけ避ける）程度で十分です。

## 30秒で言える概要（オープニング用）

「GitHub の `search/repositories` で検索し、一覧と詳細ページを `Next.js App Router` で実装しました。GitHub への通信と型の境界は `lib` に集約し、`zod` でレスポンスを検証しています。トークンはサーバーだけで使い、テストは MSW で API を差し替えています。大量件数は **ページネーション** で、API の `page` / `per_page` と URL の `page` を揃えています。」

---

## 2分で言えるアーキテクチャ

1. **画面** … トップは `/?q=&page=`、詳細は `/repositories/[owner]/[repo]`。モーダルではなく **別ページ**（課題要件）。
2. **データ** … `src/lib/github/client.ts` が GitHub REST API を呼び、**UI向けの型**に整形。Route Handler（`app/api/github/...`）も同じ関数を使い、**二重実装を避けた**。
3. **安全性** … `GITHUB_TOKEN` は環境変数で **サーバー専用**。ブラウザに渡さない。
4. **品質** … 失敗系（バリデーション、APIエラー、レート制限っぽい403、0件）を UI で区別。テストでクライアント層と主要コンポーネントを担保。

## 技術選定を聞かれたとき（要点だけ）

### Next.js / App Router

- 課題指定どおり **14以降 + App Router**。
- 一覧と詳細を **ルートで分離**しやすい。サーバーコンポーネントで **初回表示をサーバーで完結**でき、トークンをサーバーに閉じやすい。

### TypeScript

- 外部APIは仕様が変わりうるので、**境界で型を固定**し、UIはアプリ内の型だけ見るようにした。

### zod

- 「コンパイル上の型」だけでなく **実行時に壊れたJSONを弾く**ために使った。採点者には **プロダクションの防御線**として説明しやすい。

### Tailwind / shadcn 未使用

- 課題ではデザイン評価が主軸ではないため、**依存を増やしすぎない**判断。必要なら「時間があれば shadcn でコンポーネント統一」と言える。

### Vitest + RTL + MSW

- **ネットワークに依存しないテスト**が書ける。CI とローカルで再現性が高い。E2E はコスト対効果で今回は省略し、**改善余地**として言える。

### GitHub Actions（CI）

- `lint` / `test` / `build` を **提出物の再現性**のために固定した。

## ページネーションを選んだ理由（深掘り用）

**結論:** ワイヤーは「ページネーションまたは無限スクロールの配慮」なので **どちらかでよい**。こちらは **API・URL・テスト・説明責任のバランス**からページネーションを選んだ。

話すときの柱は次の4つで足りる。

1. **GitHub API がページング前提** … `page` / `per_page` と UI が素直に対応する。
2. **URL で状態が表現できる** … `?q=&page=` で共有・再表示・戻るが明快。
3. **テストとレビューがしやすい** … 「次ページ」の期待が固定しやすい。無限スクロールは重複取得・末尾検知・a11y など説明コストが上がりやすい。
4. **DOM が膨らみにくい** … 1ページ分に抑えやすい（仮想化なしでも破綻しにくい）。

### 「無限スクロールの方がモダンでは？」への答え例

「フィード型の探索には向きますが、検索結果は **ページ単位で比較・再訪**しやすい方がユーザー負荷が下がる場面があります。今回の課題スコープでは、無限スクロールに必要な **状態管理とテストの追加コスト**に見合う差分を README 上で示しにくいと判断しました。要件が『とにかくモバイルで流し読み』に寄るなら、無限スクロール + 仮想化や `IntersectionObserver` の設計を選びます。」

## 機能ごとの「なぜそう実装したか」

| 機能 | 実装の要点 | 説明の一言 |
| --- | --- | --- |
| 検索 | `GET /` のクエリ `q` | フォームは `method="get"` で **ブックマーク可能**にした。 |
| 一覧カード | オーナーアイコン + 名前（+ 補足） | 最低要件を満たしつつ、一覧での判断材料を少し足した。 |
| 詳細 | 別ルート、戻るはクエリ引き継ぎ | 課題の「モーダル禁止」に合わせ、**戻ったとき検索状態を維持**。 |
| Route Handler | `/api/github/...` | ブラウザから GitHub を直接叩かせず、**将来のプロキシ・ログ・レート制御**の拡張点にした。 |
| エラー表示 | トップ/詳細でメッセージ | レート制限は 403 をユーザー向け文言に寄せた。 |
| 画像 | `next/image` + `remotePatterns` | 外部アバターを安全に最適化。テストでは `next/image` をモック。 |

## 想定質問と答えの骨子

**Q. なぜサーバーコンポーネント中心？**  
A. トークンをサーバーに閉じ、初回HTMLで主要コンテンツを返しやすい。インタラクションが限定的な画面ではクライアント状態を増やさない方がシンプル。

**Q. なぜ GitHub をクライアントから直接呼ばない？**  
A. トークン露出のリスクと、CORS・秘匿情報の扱い。Route Handler 経由なら **サーバーだけが GitHub と話す**。

**Q. `watchers_count` は本当に意味として正しい？**  
A. GitHub REST の `watchers_count` は歴史的経緯で **star と同じ数になることがある**など、ドキュメント上の注意がある。課題要件どおりフィールドを出しつつ、面談では **APIの意味を確認したか** を正直に言う（必要なら「サブスクライバー数は別API」など補足可能）。

**Q. キャッシュは？**  
A. 今回は `no-store` 寄りで **常に最新に近い挙動**。検索の鮮度と実装単純さ優先。負荷が問題なら `revalidate` や SWR を設計する、と答えられる。

**Q. セキュリティで他にやったことは？**  
A. 環境変数をコミットしない、トークンはサーバーのみ、入力は zod で検証、エラーメッセージは内部情報を出しすぎない。

**Q. AI はどう使った？**  
A. README の「AI利用レポート」と同じ骨子で口頭化。**最終判断と検証は自分**と一言添える。
